# Claude Code 三种配置机制的设计原则

> 本文记录我在使用 Claude Code 进行团队开发时，如何区分 **CLAUDE.md**、**Commands**、**Skills** 三种机制，以及背后的设计思路。

---

## 一、三种机制概览

Claude Code 提供三种方式来约束和引导 AI 的行为：

| 机制 | 触发方式 | 加载时机 | 适合的内容体量 |
|------|---------|---------|-------------|
| CLAUDE.md | 始终生效 | 每次对话自动加载 | 短小精悍（建议 < 100 行） |
| Commands（斜杠命令） | 用户主动输入 `/xxx` | 按需加载 | 中等（操作步骤 + 参数） |
| Skills（SKILL.md） | AI 按上下文自动判断调用 | 按需加载 | 任意（可含完整模板） |

---

## 二、CLAUDE.md：始终在场的硬约束

### 本质

CLAUDE.md 的内容在**每次对话开始时自动注入上下文**，无论做什么任务都会生效。它的代价是**始终占用 token**，因此只放真正需要"时刻提醒"的内容。

### 适合写在 CLAUDE.md 的内容

**1. 无条件禁止项**

不依赖任何场景、永远不应该出现的做法。

```markdown
## 对象转换规范
禁止使用反射拷贝工具（BeanUtil.toBean、BeanUtils.copyProperties 等），统一改用 MapStruct。
```

这类规则体量极小，但如果不是"始终提醒"，AI 偶尔会遗忘或打折扣执行。

**2. Skill 路由索引**

当项目有多个 Skill 时，CLAUDE.md 承担"目录"职责，告诉 AI：遇到什么情况应该查阅哪个 Skill。

```markdown
## Skill 索引
| Skill 名称 | 关键词 |
|-----------|--------|
| design-doc-required | 写代码前、新功能、bug 修复 |
| git-commit-standards | 提交、commit、git |
```

这样 AI 不需要每次都扫描所有 Skill 文件，通过索引快速定位后按需加载。

**3. 项目级全局规范（极简形式）**

仅保留最核心的一两条"方向性原则"，细节放到 Skill 里。

### 不适合写在 CLAUDE.md 的内容

| 不适合的内容类型 | 原因 |
|----------------|------|
| 详细的操作步骤（超过 10 行） | 每次对话都消耗 token，大多数任务用不到 |
| 文档模板 | 体量大，只在特定场景需要 |
| 完整的流程规范（如提交规范的所有细节） | 只在 commit 时才需要，不应始终占用上下文 |
| 有条件的规则（"当 XX 时才适用"） | 有条件说明不是"始终需要"，应放 Skill |

### 判断标准

> **如果一条规则在 99% 的对话中都应该生效，放 CLAUDE.md。否则放 Skill 或 Commands。**

---

## 三、Commands：用户主动触发的工作流

### 本质

Commands 是存放在 `.claude/commands/` 下的 Markdown 文件，用户输入 `/命令名` 时触发。**由人决定何时调用**，AI 不会自动判断是否需要执行。

### 适合写成 Commands 的内容

**1. 需要参数的重复性操作**

```
/new-feature payment-refund
```

Commands 支持 `$ARGUMENTS` 占位符，适合模板化的创建类任务（新建功能骨架、生成 CRUD 代码、初始化模块结构等）。

**2. 用户有明确意图才执行的操作**

例如"为当前分支生成 Release Note"、"扫描并列出所有 TODO 注释"——这类操作不应由 AI 自动触发，必须等用户主动发起。

**3. 跨项目通用的脚手架操作**

放在用户级别的 `~/.claude/commands/` 下，在所有项目中都可以调用，不绑定具体项目。

### 不适合写成 Commands 的内容

| 不适合的内容类型 | 原因 |
|----------------|------|
| "必须在某动作前执行"的流程门禁 | 用户可能忘记手动触发，应改用 Skill 自动拦截 |
| 纯规范约束（不产生操作） | 规范不是命令，放 CLAUDE.md 或 Skill |
| 需要 AI 读取上下文再决定是否执行的逻辑 | Commands 是无条件执行，不带判断；有条件判断用 Skill |

### 判断标准

> **如果是"用户意识到需要某个操作才会触发"，用 Commands。如果是"AI 应该在特定情境下自动介入"，用 Skill。**

---

## 四、Skills：AI 自动判断调用的流程门禁

### 本质

Skills 是带有 `description` frontmatter 的 Markdown 文件。Claude Code 根据 description 中的触发条件，在对话过程中**自主判断**是否需要调用。**由 AI 决定何时调用**，无需用户输入命令。

核心价值在于：**把流程约束从"靠人记住"变成"由 AI 强制执行"**。

### 适合写成 Skills 的内容

**1. 流程门禁（Before 类规则）**

在某个操作发生之前，AI 必须先完成某件事。

```yaml
description: |
  Use when about to write any implementation code.
  MUST invoke this skill BEFORE writing a single line of code.
```

例如：写代码前必须有设计文档、提交前必须分析 diff、写文档前必须读索引。这类规则如果放在 CLAUDE.md，AI 偶尔仍会跳过；放成 Skill 后，因为 description 里明确写了触发条件，Claude Code 会在匹配时强制调用。

**2. 包含大量细节的规范**

阿里巴巴 Java 编码规范（命名、注释、异常、并发……），内容超过 500 行。这些内容只在写 Java 代码时需要，不应常驻上下文，非常适合按需加载的 Skill。

**3. 带模板的文档规范**

Bug 分析文档、设计文档有固定的章节结构和模板。模板本身体量较大，通过 Skill 在需要时加载，比写死在 CLAUDE.md 里更合理。

**4. 需要 AI 做判断后再执行的多步流程**

```
读取现有文档索引
    → 分析内容边界
    → 判断是新建还是补充
    → 执行写作
    → 更新索引
```

这种"先思考再执行"的流程，通过 Skill 的详细步骤说明来引导 AI，比 Commands 的无条件执行更合适。

### 不适合写成 Skills 的内容

| 不适合的内容类型 | 原因 |
|----------------|------|
| 一句话的硬禁止规则 | 体量太小，且需要始终生效，放 CLAUDE.md |
| 用户有意识触发的操作 | 用户知道自己要做什么，用 Commands 更直接 |
| 与其他 Skill 高度重叠的内容 | 合并到已有 Skill，避免触发条件冲突 |

### 判断标准

> **如果是"AI 应该在特定情境下主动介入、且内容较为复杂"，用 Skill。**

## 五、三者的核心区别：谁来决定触发

```
┌─────────────────────────────────────────────────────────────┐
│                    触发决策者                                 │
├─────────────────┬───────────────────┬────────────────────────┤
│   CLAUDE.md     │    Commands       │       Skills           │
│                 │                   │                        │
│  无需决策       │  用户决定         │  AI 决定               │
│  始终生效       │  /命令名 触发     │  上下文匹配后自动调用  │
└─────────────────┴───────────────────┴────────────────────────┘
```

这一维度决定了内容的分配逻辑：

- **不信任"会被记住"的规则** → CLAUDE.md（强制始终在场）
- **不信任"AI 会主动做"的操作** → Commands（用户主动触发）
- **不信任"用户会记得在正确时机触发"的流程门禁** → Skills（AI 自动判断介入）

## 六、在本项目中的实践

### 全局 CLAUDE.md（`~/.claude/CLAUDE.md`）

存放跨项目的个人偏好：Git 提交署名、个人编码习惯等。内容极简，不超过 20 行。

### 项目级 CLAUDE.md（`项目根/CLAUDE.md`）

存放两类内容：
1. 该项目的硬禁止规则（如"禁止使用 BeanUtils"）
2. 已安装的插件/Skill 路由索引（让 AI 知道去哪里找详细规范）

以 caseflow 插件的 CLAUDE.md 为例，它本身就是一张 Skill 索引表，不含任何细节规范——细节都在各 Skill 文件里。

### Skills（`caseflow` 插件）

| Skill | 触发条件 | 为什么不放 CLAUDE.md |
|-------|---------|---------------------|
| `design-doc-required` | 写任何实现代码前 | 包含完整文档模板，体量大 |
| `bug-doc-required` | 编写 bug 分析文档时 | 包含章节结构约束和 Mermaid 要求 |
| `java-coding-standards` | 编写/审查 Java 代码时 | 规范内容超过 500 行 |
| `git-commit-standards` | 执行 git commit 前 | 只在提交时需要，且需读取 diff 后执行 |
| `pre-implementation-code-orientation` | 文档写完、开始实施代码前 | 需要 AI 读文档提取坐标后才能执行 |
| `doc-index-required` | 写 docs/ 下任何文档前 | 需要先读索引再判断，是多步流程 |

### Commands（暂未使用，以下为规划示例）

如果未来需要脚手架类操作，适合做成 Commands：

- `/new-module {模块名}` - 按模板初始化新业务模块目录结构
- `/gen-mapstruct {源类} {目标类}` - 生成 MapStruct Convert 接口骨架

## 七、一句话决策树

```
新的约束/规范/操作需要落地
        │
        ├─ 是"永远不能做"或"永远要做"的硬规则，且内容极简？
        │       └─ 是 → 写入 CLAUDE.md
        │
        ├─ 是"用户主动发起"的操作，可带参数？
        │       └─ 是 → 写成 Commands（.claude/commands/xxx.md）
        │
        └─ 是"AI 应该在特定情境下自动介入"的流程？
                ├─ 规则简单（1-2 句话）且需始终生效 → 写入 CLAUDE.md
                └─ 有复杂步骤 / 含模板 / 需 AI 判断 → 写成 Skill
```
