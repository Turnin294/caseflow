# 开发日志 · 2026-04-23

## 17:00 · 修改 Skill · korepos-backend-service

**原因：** 原 skill 把"跨模块访问一律走 BackendInfra 门面"解读得过严，导致
`features/refund/backendv2/service/*` 想直接调 `features/payment/backend/service/KPayOnlineRefundService`
时被判违规，只能强行在 common 层引入 outcome value object + 门面映射方法做封装，
引入不必要的胶水代码。实际语义上这两者都属"后台团队介入开发的代码区"，
独立服务化时会一起搬走，不存在 ACL 隔离的必要。用户明确要求放开：backend 模块之间可直接互引，
只约束对"非 backend 层（UI 团队维护的 application / data / domain / presentation）"的跨模块访问。

**改动：**

- SKILL.md §核心原则：从"跨模块访问只能通过门面"改为按层区分（backend 层直通 / 非 backend 层经门面）
- SKILL.md §引用边界：开头补"核心心智模型"两条（backend 阵营互通 / 非 backend 层是禁区）
- SKILL.md §允许引用清单：新增 `lib/features/{other}/backend/**`
- SKILL.md §禁止引用清单：原 `lib/features/payment/**`、`cart/**`、`bill/**` 粗粒度条目改为精确的
  `lib/features/{other}/{data,application,presentation,domain}/**`
- SKILL.md §"发现越界 import 时立刻停下"补分支处理：backend 层直 import / 非 backend 层走门面扩展
- SKILL.md §BackendInfra 门面规则开头句加限定：只对"非 backend 层的跨模块依赖"强制走门面
- SKILL.md §新增跨模块依赖流程重写为情况 A/B 两分支（A 其它 backend 层 → 直接 import；B 非 backend 层 → 走门面三步）
- SKILL.md §禁区表第 4 条（直接 import other presentation）改为"其它 feature 的非 backend 层"，
  并明确"其它 feature 的 `backend/` 层不受此限"
- plugin.json / marketplace.json version: 1.10.1 → 1.10.2（Patch，规则措辞调整）

**影响：**

- 今后 backendv2 / payment-backend / refund-backend 之间可直接 import service / dao / dto，无需 BackendInfra 中转
