# 开发日志 · 2026-05-04

## 09:40 · 修改 Skill · backend-knowledge-graph-required

**原因：** 后端项目对话中经常会反复提到某个业务、表、字段来源和查询逻辑，但之前图谱只强调表逻辑和原子能力，没有把 SQL 查询逻辑作为一等资产自动归档，导致“这个业务怎么查 / SQL 怎么完善 / 对应哪些表和原子能力”仍容易散落在会话里。

**改动：**
- 将 SQL 查询逻辑纳入 `backend-knowledge-graph-required` 的强触发范围
- 新增 `_sql_candidates.md` 候选池，要求会话中提到 SQL / DAO / Mapper / 查询逻辑时自动记录 SQL 指纹
- 新增 `09_sql_query_index.md` 和 `sql-queries/{business-scenario}.md` 规范，用于沉淀项目原子能力对应的 SQL
- 强化全景 ER 归档：整理 SQL 时同步合并到 `02_data_model_map.md` / `tables/`，形成表关系视角
- 补充 SQL 指纹去重合并规则，避免同一业务查询因为 where 变体重复建卡

**影响：**
- 后续问业务查询逻辑时，AI 必须把 SQL、表关系、过滤/聚合/排序语义、原子能力和代码坐标一并沉淀
- 新增接口前可以先查 SQL 索引判断已有查询和原子能力是否可复用
