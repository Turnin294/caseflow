# 开发日志 · 2026-06-15

## v1.35.0：注释红线机械兜底 `check-comment-density` 默认从 `warn` 反转为 `block`

**原因:** 用户在某 Java 项目里发现模型写出 `// PM-IT-2607 6.6 dye factory delivery time`——工单号 + 英文注释，同时违反 `coding-standards-common` §5.4（禁工单 / Ticket / 内部流水号引用）与 §5.0（注释用当前会话沟通语言，中文沟通应写中文）。用户追问："这是模型的问题还是注释插件的问题？护栏不应该在 plugin 里统一做好么，要用通用插件来约束，而不是各自写一套。"

排查结论分两层：(1) 责任在模型——它写出了违反既有规则的注释，插件规则本就明文禁止，非插件"导致"。(2) 但 `check-comment-density.js` 的 `ticket-code` 正则实测能命中 `PM-IT-2607`，hook 却默认 `warn` 模式（exit 0，仅 stderr 提示，不阻断），所以模型照样落盘。即"规则在、检测在，差最后一脚硬拦"。warn 默认是 v1.29 引入时为"试用期评估误报率"而设的临时档位。

**改动:**
- `hooks/check-comment-density.js`：默认模式 `warn` → `block`。客观红线（变更标记 / 日期 / 工单号 / 带元信息分节线 / 版本流水措辞）命中即 `exit 2` 阻断。
- **`long-block`（连续注释块超阈值）单独保留为启发式软规则，只提示不阻断**：用 `findings.some(f => f.id !== 'long-block')` 判定是否有"硬规则"命中，只有硬规则命中且处于 block 模式才 exit 2。理由——long-block 是行数启发式，公开 API 的合法长 dartdoc 容易撞阈值，硬阻断会误伤正常文档；而工单号 / 日期 / 标记等是客观无歧义项，误报率极低，适合硬拦。
- 降级路径反转：`TEAM_STANDARDS_COMMENT_HOOK=warn` 降级为仅提示、`=off` 关闭（原先是 `=block` 升级）。
- 测试：`tests/check-comment-density.test.js` 更新 1 条（[BUGFIX] 默认行为从 exit 0 改为 exit 2），新增 3 条（默认 block 下 long-block 单独命中不阻断 / 客观红线 exit 2 / warn 降级仅提示）。30 条全过。
- 同步文档：§5.4 机械兜底注、`hooks.json` `_comment`、CLAUDE.md / AGENTS.md / README.md 辅助资源表。

**影响:**
- 装了插件即强制：注释红线不再依赖会话是否触发 skill，也不依赖每个项目各自在 settings 写 env——护栏在插件里统一做死，这正是用户要的"通用插件约束"。
- 误伤面收敛：只有客观红线硬拦，long-block 仍是软提示，合法长 dartdoc 不被挡。
- 向后兼容降级：个别项目若需放宽，本会话设 `TEAM_STANDARDS_COMMENT_HOOK=warn`/`=off` 即可，不改插件。
- 注意：仅反转 `check-comment-density` 一个 hook 的默认值；`check-backend-kg-readiness` 仍保持 `warn` 默认（不在本次范围）。
