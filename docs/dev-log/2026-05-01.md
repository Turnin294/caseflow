**影响：**

## 11:05 · 沉淀核心红线 · korepos-backend-service

**原因：** 用户在 refund_price_service 兜底分摊改动中发现 service 层频繁出现裸 SQL（`db.customSelect('SELECT ...')`），把数据访问逻辑混在业务编排里。skill 里 Step 4 虽强调 DAO 是 SQL 容器，但缺少对偶规则——没有显式说明「Service 层一行 SQL 都不能出现」，导致历史代码反复违规。需把这条核心红线沉淀到 skill。

**改动：**

- 在 `korepos-backend-service` Step 5「强制规则」节新增核心红线：**Service 内禁止任何 SQL**——`_db.customSelect` / `_db.select(table)` / `_db.update(table)` / `_db.delete(table)` / `into(table).insert(...)` / `_infra.db.batch(...)` 等任何 drift 直接读写形式均不允许；`db.transaction(() async { ... })` 是唯一例外但事务体内只能调 DAO 方法
- 在 Step 4 头部追加引言，明确 DAO 是 backend 内 SQL 的**唯一容器**（含 service / orchestrator / handler / registry / internal 全部禁用 drift 调用）
- 「禁区」节追加对偶条目，与 Step 5 红线呼应；附违规处置：发现 service 层 SQL → 立刻下沉到 `backend/dao/` 或 `common/backend_infra/daos/` 原子方法 + 登记 `docs/coding-violations.md`
- 「参考范本」节新增「最佳实践活范本」小节，以 `/confirm/refund/transaction` 链路 5 文件（endpoint 枚举 / Request DTO / Response DTO / Service / DAO 注入与调用）作为新接口"参照拷贝"样板，列明各文件关键看点与"不要拷贝"项（backendv2/ 这个名字、backendv2/dto/ 这个 DTO 目录两者均属历史遗留）
- 同步更新 `CLAUDE.md` 的 `korepos-backend-service` 索引行覆盖范围与关键词

**影响：**

- AI 写新接口时，service 层一行 SQL 都不能出现；遇到旧代码裸 SQL 在改同段代码时顺手下沉到 DAO
- 新接口 endpoint / request / response / service / dao 直接对照 `/confirm/refund/transaction` 链路 5 文件落盘，减少自行推断带来的形态漂移
- coding-violations.md 与 skill 形成闭环：违规登记后下次编码自动回顾，避免同类重犯

## 12:30 · 沉淀核心原则 · architecture-ddd-lite-fullstack

**原因：** 用户在订单数据同步链路 bug（`order_item_payment_allocateData` 漏接收）的治本过程中，发现 AI 把 95 行新增 record 处理逻辑直接塞到了旧 1500 行巨型方法 `OrderRepository.pullOrderData` 内部——典型的"在旧代码就地堆叠"反模式。正确做法是把新代码放到新结构（独立子门面 `OrderSyncService` 暴露 public 方法 `persistPaymentAllocates`），旧代码只 +1 行调用。这种"新代码落点决策"是默认行为级别的架构原则，需要沉淀到 skill 防止重犯。

**改动：**

- 在 `architecture-ddd-lite-fullstack`「结构质量门禁」节新增子节「新代码落点决策（扩展现有功能时不要在旧结构堆叠）」：
  - 默认原则：扩展能力时新代码放到新结构暴露 public 方法，旧代码只 +1 行调用
  - 给出"为什么是默认行为而不是可选优化"的对照表（旧文件污染面积 / 是否符合目标态 / 重构成本 / 单测粒度 / 调用方迁移路径 5 维度）
  - 触发场景清单（bug 修复 / 功能扩展 / 数据同步链路补字段 / 即将动巨型方法）
  - 5 步正确决策流程
  - 反模式硬约束："即使旧代码本来就乱也不允许跟着抄"
- 「禁止行为」表追加红线："在已有的巨型方法 / 旧骨架文件里就地追加新逻辑"
- `korepos-backend-service`「禁区」表对应追加一条 backend 场景红线，引用 `architecture-ddd-lite-fullstack` 的「新代码落点决策」节
- 更新 `CLAUDE.md` `architecture-ddd-lite-fullstack` 索引行覆盖范围与关键词

**影响：**

- AI 扩展现有能力前必须先做"新代码落点决策"——独立可命名 → 新结构暴露；不能因为"旧代码已经这样了我跟着抄"就在巨型方法里堆叠
- 历史巨型方法 / 老骨架文件不再被新代码继续污染，后续重构时只需迁移既有的"旧实现部分"，不会产生"新增的旧风格代码"这种持续债
- 与 strangler pattern 路线对齐：新结构稳定后调用方可立刻切换，不需要等旧代码删除
