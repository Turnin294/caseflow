**原因:** 用户在 korepos `refund_confirm_service.dart` 上发现 AI 反复违规注释规则——私有方法 `_validateMethodsAmountSum` 上方写 12 行 dartdoc 讲"前端契约 / 早期版本曾要求 / 现已统一"的契约迁移史;`if (cancelBridgeServiceFeeAmount != null)` 上方堆 5 行行内 WHY 讲"另一分支怎么走 / 联台 scaleRatio / 不走本分支不受影响";section divider `// ===== [ADDED 2026-04-28] xxx =====` 风格分节标题带日期。用户原话:"代码内部不需要一堆废话注释 / 一堆分化注释主要说明函数能力和参数就行 / 每次改两个地方不友好吧"。

## v1.29.0:新增第五道 hook `check-comment-density.js` 为 §5.4 注释红线做机械兜底

**原因:** 同一会话内,AI 在 korepos `pending_online_refund.dart` 表头给索引补注解时,又写了 4-6 行 `///` doc comment 把"注解只对 onCreate 生效 / 存量设备靠 migrateToVersion12/15 补 / 两者并存幂等"这种迁移共存史塞进源码。用户当场质问:"怎么写个代码又出现这么多注释 / 我的 plugin 没约束成功么"。排查结论:§5.4 注释红线此前**只在 skill 文档里**,纯靠模型自觉触发 `coding-standards-common`;`Write|Edit` 那道 PreToolUse hook 只管 DTO 注解 / 设计文档 / 后端知识图谱,**没有一条管注释**。会话漏触发 skill 时,红线形同虚设。

**改动:**
- 新增 `hooks/check-comment-density.js`(PreToolUse Write/Edit/MultiEdit,默认 warn):只扫本次**新增文本**(Edit.new_string / MultiEdit.edits / Write.content),去字符串字面量后判定注释片段,命中即 stderr 提示。抓客观无歧义项:变更标记 / 注释日期 / 工单·PR·Issue 号 / 带个人或日期 TODO / 带元信息分节线 / 版本流水措辞,外加连续注释块超阈值(默认 6)软提醒。

3. **hook 抓不全,要在规则源写清边界。** 本次用户被吐槽的具体注释(迁移共存 prose)hook 默认阈值其实抓不到——它不含标记/日期,是叙事。§5.4.1 明确"prose 式实现史仍靠 skill + 评审",避免给人"装了 hook 就不会再漏"的错觉。

## v1.30.0:新增 `comment-cleanup` skill —— 存量注释批量清理

**原因:** v1.29.0 的 `check-comment-density` hook 落地后,在 korepos `pending_online_refund.dart` 上立刻暴露两个缺口:(1) hook 只 warn **新写入**内容,对存量满是 `v13 新增 / v14 起 / v12 及以前` 的老文件束手无策;(2) 实测 hook 连 `v13 新增` 都漏抓——version-flow 正则只覆盖「流水/调整/历史」没覆盖「新增/起/及以前」这类最常见中文版本标记。而 §5.5 只规定「改到哪清到哪」+「大量历史垃圾单独开 PR」,没有执行那条「单独清理」的标准流程。用户提议补一个「清理存量注释」的能力,借鉴开源最佳实践、多语言。

**方案选型(经 solution-review):** 候选有「独立 skill / 仅 slash 命令 / 给 hook 加 scan 模式 / 不加只补 hook」。关键判断:**批量清理需要语义判断**(prose 实现史、契约演变史 regex 抓不到),不是 hook 能干的;§5.5 已认账「单独清理」但缺 workflow。用户拍板「独立 skill」+「补 hook 正则」。

**改动:**
- 新增 `skills/comment-cleanup/SKILL.md`:用户主动触发,对存量文件/类/模块成批清理;多语言按扩展名识别注释符;红线规则**单一来源引用 §5.4 + §5.4.1**,本 skill 只管范围圈定 / 分类决策(删·改写·保留·待定)/ 安全边界(只动注释不动逻辑、待定项必问、不扩范围)/ 提交纪律(单独 commit、不夹带逻辑)。
- `hooks/check-comment-density.js` version-flow 正则补 `新增/引入/上线/移除/删除/废弃/弃用/起/及以前/及以后`;测试 23→27 例(含 `v3 协议`/`IPv6 上线` 不误报)。
- `CLAUDE.md`/`skill-flow.md`/`README.md`/`AGENTS.md` 登记新 skill,skill 计数 24→25;skill-flow FAQ 新增「三套注释机制(§5.5 顺手 / hook 写入拦截 / comment-cleanup 存量批量)分工」。
- 三处 manifest 1.29.0 → 1.30.0(MINOR,新 skill)。

**决策原则沉淀:**
1. **同一份红线、三个触发面,规则只写一处。** §5.5(顺手)/ hook(写入新内容)/ comment-cleanup(存量批量)分工不同,但「什么注释该删」全部引用 §5.4,谁都不重定义,避免三地漂移。
2. **hook 与 skill 各司其职:机械的归 hook,语义的归 skill。** `vN 新增` / 标记 / 日期这类确定字面下沉 hook;prose 实现史 / 契约演变史这类要读懂上下文的归 skill。本次同回合发现 hook 正则漏 `vN 新增`,顺带补——但补完它仍只是「机械兜底」,真正成片清存量还得靠 comment-cleanup 的语义判断。
3. **清理是按需动作,但仍做成 skill 而非纯命令。** 用户选独立 skill;为防自动触发误清,skill 定位为「用户主动要求才触发」+「待定项必问 + 只动注释不动逻辑」双保险,不做编辑后自动销毁式清理。
