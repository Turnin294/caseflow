# 开发日志 · 2026-05-03

## 16:30 · 修改 Skill · design-doc-required

**原因：** 在 BFF 项目（kpay-pos-business-app-bff）实施一次极简改动时——给 `pullOrderData` 透传层 DTO 补 2 个上游 SDK 已存在的字段 + Service 加 2 行 setter 映射，共 4 行净变更——按现有规则触发了「轻量模版」分支生成接口级设计草稿。用户当场反馈：「这种简单的修改草稿都不需要 因为 git commit 足够说明变更」。问题暴露在原规则的边界：第一·七步轻量模版准入硬清单虽然挡住了"新建表/跨服务/新中间件"等大改造，但对"BFF 透传层字段补漏 / 局部修正 / dead code 清理 / 注释整理"等改动仍然要求生成 mermaid 流程图和填写第 1-5 节，文档成本明显高于收益。原「合法的例外情况」只列了"测试/配置/重构/拼写修正"四条，没有覆盖「业务源码极简改动」这类高频场景。需要把"git commit body 即变更说明"沉淀为 skill 级第三档合法例外，并给出明确硬清单避免被滥用。

**改动：**
- `skills/design-doc-required/SKILL.md` § 合法的例外情况 新增第 5 条「极简代码调整」
- 同节新增「极简改动硬清单」「极简分支处理动作」「极简分支判定边界示例」「红线」「与轻量修订和完整流程的区别」5 个子章节
- 极简硬清单（必须全部 ✅）：≤2 文件 ∧ ≤30 行净变更 ∧ 0 新类/数据库表/字段/对外契约入口 ∧ 既有出参字段语义不变（仅新增、向前兼容）∧ 无业务规则/状态机/中间件/锁/事务改动 ∧ 不跨模块/服务 ∧ 改动性质属于（透传层补漏字段 / 局部行为修正 / 简单条件分支调整 / 移除 dead code / 注释 import 整理）之一
- 极简分支处理：不生成草稿、不生成 coding.md、不更新设计文档；commit body 必须承担变更说明（改了什么 + 为什么改 + 影响范围）
- 执行流程图新增 TRIVIAL 前置判定节点：「改动通过极简改动硬清单?」→ 是则 commit body 即变更说明直接进入实现，否则进入原查找设计文档流程
- 红色警告表新增 2 条防线：「用户说简单改动不需要文档」「极简分支跳过了文档，commit body 也可以简单写」
- `CLAUDE.md` Skill 索引表 design-doc-required 行的「覆盖范围」与「关键词」更新为三档分级（极简跳过 / 轻量 / 完整）
- `docs/skill-flow.md` 头部新增 v19.9 变更摘要
- `.claude-plugin/plugin.json` 与 `.claude-plugin/marketplace.json` 版本 1.18.7 → 1.18.8

**影响：**
- 透传层补漏字段、≤30 行局部修正、dead code 清理等极简改动不再被强制要求轻量草稿，commit body 承担文档职责
- 边界明确：硬清单任意一项 ❌ 立即退回轻量/完整模版，不允许凭"看着简单"绕开
- 极简分支的代价是 commit body 必须写清楚改了什么 + 为什么改 + 影响范围，否则等于无文档无说明
- 与现有「第四·五步轻量修订」（设计文档已存在的小修）形成互补：极简跳过用于无设计文档的极简改动，轻量修订用于有设计文档的小修
- 用户实操痛点（"BFF 补 2 字段也要写文档"）从此有规则级解法
