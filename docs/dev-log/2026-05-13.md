## 背景

v1.26.0 把 P1 + P2 的 7 项结构优化一次性合并后, 远程专家审计发现 5 处文档漂移:

1. README / CLAUDE / AGENTS 还写 `check-design-doc.{cmd,sh}` 默认禁用, 但 hooks.json 实际已经默认启用了重写的 `check-design-doc.js`
2. CLAUDE.md 三档表说 S 档"不需要调 design-doc-required", 但 hook 在无设计文档项目里仍会拦截 —— 语义上有冲突
3. README 把 `check-design-doc.js` 描述成"完整设计门禁", 实际它只做"项目级存在性兜底", 不验证"本次需求对应文档"
4. `docs/skill-flow.md` 子循环图说"默认生成用户目录草稿, 不更新 INDEX.md", 与 v1.20 起的"用户目录与 docs 同等执行 Phase-A/B"规则冲突
5. `docs/skill-dependencies.md` 还是"全链路依赖图"视角, 没体现 v1.26 引入的 S/M/L 档位分流与沉淀类批处理

## 决策原则沉淀

1. **hook 默认启用要在 5 处文档同步**: README 顶层目录表 / README 关键文件表 / README 拦截章节 / CLAUDE.md 辅助资源表 / AGENTS.md(sync-agents 自动)。少了任何一处, 用户看 README 与实际跑出的拦截行为对不上, 是最容易被吐槽的"文档漂移"。
2. **强门禁的语义边界要写在文档里, 不只在代码里**: `check-design-doc.js` 是"项目级存在性兜底"还是"完整设计门禁"差很大——前者 hook + skill 协同, 后者 hook 就是单点 SPOF。专家审计点出这一条就是因为 README 描述把它说大了。语义边界写到文档, 是给用户(包括未来的 AI)的契约。
3. **S 档跳过 vs hook 阻断的协同**: 跳过 skill 工作流程 ≠ 跳过 hook 兜底。这种"两套规则交叉作用"的场景必须用对照表展示三类典型情形(已有文档 / 新项目 / M-L 档), 不能只靠文字描述。专家踩这一脚, 反映出 AI 在执行时确实会被"规则表说跳过, hook 说阻断"两边拉扯。
4. **档位是 v1.26 的核心结构性变化, 所有依赖类文档都要同步**: skill-flow / skill-dependencies / skill-triggers 三份文档形成一个三角, 任一份脱钩就会出现"AI 看 A 文档执行 X, 看 B 文档执行 Y"的内部矛盾。本次补 skill-dependencies, 让三份对齐。

## v1.26.2 防回归补丁: CI 自动化守卫 + hook 注释一致 + 升级提醒

紧接 v1.26.1 文档补丁,远程专家又发现 5 处偏"防回归"的工程化加固点。本次都是给 CI 加自动检查、删 hook 注释里的过时说明、补 README 升级提醒,无规则方向变化。

### 本次变更

| 编号 | 变更 | 文件 |
|---|---|---|
| **G1** | 新增 `scripts/check-version-sync.js` 守住三处 manifest 版本一致 | `scripts/check-version-sync.js`(新) + `.github/workflows/ci.yml`(新 job `version-sync-check`) |
| **G2** | `check-cross-refs.js` 扫描目标扩到 31 个文件(原 28) | `scripts/check-cross-refs.js` 加入 `docs/skill-dependencies.md` / `docs/skill-triggers.md` / `docs/anti-pattern-case-library.md` |
| **G3** | `check-design-doc.js` 注释行删除过时说明 | `hooks/check-design-doc.js`:旧注释"创建新文件且文件不存在(允许 design-doc-required skill 走完正常流程后再 Write)"实际代码无此分支, 删掉换上"源码新文件同样进入检查"说明 |
| **G4** | README 升级章节加 v1.26 hook 默认启用提醒 | `README.md` 升级段后新增"⚠️ 从 v1.25.x 或更早版本升级到 v1.26+ 的注意事项"小节, 列三类项目情况的处置 |
| **G5** | `audit-skills.js` 加 `--ci` 退出码 + CI job 守卫 | `scripts/audit-skills.js` 加 `--ci` flag(有警告 exit 1) + `.github/workflows/ci.yml` 新 job `audit-skills-check` |

CI workflow 从 3 job 扩到 **5 job**:hook 单测矩阵 / sync-agents / cross-refs / version-sync / audit-skills。

### 决策原则沉淀

1. **手动维护的不变量必须有 CI 守卫**: 版本号同步、SKILL.md 健康度、跨文档锚链解析这三件事都是"漏掉一处就影响用户"的不变量。专家两轮审计都点版本号 / 文档漂移 / audit 警告, 共同信号:**人工维护不可靠, 必须把检查写进 CI 才能止漏**。本次三个都补上后, 后续发版漏一处版本号 / skill 涨到 1000+ 行 / docs 改章节名忘改引用, 都会被 PR 检查拦下。
2. **hook 注释与实现的语义一致也是工程债**: `check-design-doc.js` 注释里写"创建新文件不存在则放行"但代码实际无此分支, 这种类型的"代码-注释不一致"在 hook 这种关键节点危害极大——后续维护者会按注释逻辑改代码、引入真实 bug。删除过时注释比保留更安全。
3. **升级提醒要写在 README 升级章节,不能只在 hook 文档**: v1.26 启用 check-design-doc.js 这种"默认行为变更"必须在 README 升级路径上显著标注, 否则老用户从 1.25.x 升上来第一次写源码被拦, 会以为是 bug。这是版本迁移文档的标配。
4. **CI job 阈值要保守、可调**: `audit-skills.js` 当前阈值是 description 800 字 + SKILL.md 800 行 + 0 引用 + 半年 stale。其中 800 行是经过 v1.26.0 拆分实测验证的, 多数 skill 拆完都在 500 以下;一旦未来需求把某 skill 拉到 800 以上, CI 直接拦, 强制要么拆 rules/ 要么调阈值, 不能默默涨。

### 后续维护规则

- 任何手动维护的"多处必须一致"的不变量, 先考虑能不能写检查脚本而不是写文档约束
- `audit-skills.js --ci` 的警告阈值后续可调, 但调高(放松)需要在 dev-log 里写明理由
- `scripts/check-version-sync.js` 后续如果增加第 4 个 manifest(如 npm package.json 也要带版本号), 在 `targets` 数组追加即可
