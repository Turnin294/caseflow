# 开发日志 · 2026-04-22

## 修改 Skill · korepos-backend-service（补充 ACL 防腐蚀层规范）

**原因：** 用户在 korepos-refund 修复 `confirmRefund` 响应 `kposRefundTransactionList` 为空 bug（backendv2 v6）时发现两个规约空白：

1. 调试业务逻辑时需要给 DAO 产出的内部 DTO 加字段（如 `originalPayChannelCode`），但该 DTO 被 `typedef` 复用为 `/v2/refund/write` 的 HTTP 响应，直接加字段会把内部字段泄漏到 wire JSON；虽然 freezed `fromJson` 默认忽略未知字段不会挂，但仍是对已对接契约的污染
2. 现有 skill 没有规定：已对接接口（UI 已按 §4.N 写调用代码）的 DTO 在后续调试/迭代时哪些改动合法、哪些禁止；用户原话："自己改业务相关逻辑的时候不要影响到 http 的返回 尽量 接口已开启对接 所以写好的接口做好防腐设置"

**改动：**

- `skills/korepos-backend-service/SKILL.md`：
  - 在「DTO ↔ UI 对接手册双向绑定与同步」与「完成后自检清单」之间新增一整节 **「ACL（Anti-Corruption Layer）：内部 DTO 与 wire DTO 的边界」**，包含：
    - 背景与原则（internal-first 决策原则）
    - 三档 ACL 策略（L1 `@JsonKey` 注解豁免 / L2 双 DTO 或双枚举物理分离 / L3 mapper 层）
    - 判断「wire vs internal」的三问决策
    - L1 代码样例（`@JsonKey(includeToJson: false, includeFromJson: false)`）
    - L2 代码样例（引用项目内现有实例 `PaymentType` internal vs `RefundMethodType` wire）
    - 已对接接口保护矩阵（7 种改动分类的允许/禁止规则）
    - AI 自检判断流 Mermaid 图
    - grep 识别「是否已对接」的标准步骤
  - 「完成后自检清单」表追加两行：ACL 分级标注 / 已对接接口保护
  - 「禁区」表追加两行：未加 `@JsonKey` 让 internal 字段进 wire / 对已对接接口做破坏性改动
- `.claude-plugin/plugin.json`：version `1.9.1` → `1.9.2`（Patch，内容扩充）
- `.claude-plugin/marketplace.json`：version 同步 `1.9.1` → `1.9.2`

**影响：**

- 今后在 korepos / korepos-refund 系列项目编写或修改 backend DTO 时，AI 必须按「三问决策」判定字段归属；internal 字段默认走 L1 注解豁免
- 改动已对接接口的 DTO 前，AI 必须先 grep `presentation/` / `frontend/` 确认对接状态，再按「已对接保护矩阵」判断本次改动合法性
- 本次变更属于 skill 内容扩充，未改动 skill 链路结构，未创建 skill-flow 版本快照
- `CLAUDE.md` 主索引表不含 korepos-backend-service（专项 skill），无需同步；`README.md` 不含该 skill，无需同步
- 本次回填的 ACL 规则同步指引了当下 korepos-refund backendv2 v6 的修复方向（`RefundTransactionResult.originalPayChannelCode` 按 L1 加注解的建议）

## 新增 Skill · daily-work-log（业务项目每日工作日志）

**原因：** 用户希望在业务项目修改代码时自动沉淀「今天做了什么」—— 原话：「再补充一个 skill 每次修改代码调整代码的时候给出记录 一个 bug 一个功能 区分一下，一行行记录，这个记录后期会作为每日工作日志，写入的时候读取一下 同一个 bug 修复描述到一个问题中，同一个功能开发也记录到一个问题中，新功能开发和 bug 调整可能记过多轮代码修改 把修改明细也记录一下并且补充一个合理的预估消耗工时」。

现有 `dev-log` 只覆盖 caseflow 仓库自身的变更，**不涉及业务项目**；业务项目源码改动后只有 `git log` 能追溯，但 git log 按提交而非按主题聚合，无法回答"今天修了哪几个 bug 共花了多少工时"的日报问题，规约空白。

**改动：**

- **新增 Skill** `skills/daily-work-log/SKILL.md`：
  - 触发时机：业务项目源码 Edit/Write 后 / 用户说「记录一下」「更新工作日志」 / 会话结束前强制回补
  - 文件路径：业务项目 `docs/work-log/{YYYY-MM-DD}.md`
  - 分两区：`🐛 Bug 修复` / `✨ 功能开发`，条目 ID 按日递增 `[B1][B2]...` / `[F1][F2]...`
  - 写入前 Step 0 必须 Read 当天日志，提取已有条目标题做合并判定（bug 文档标题 / 设计文档标题 / 用户指定主题）
  - **修改明细一行一条**：带 `HH:MM` 时间戳、动词开头（改/加/删/抽）、对象具体、≤ 50 字
  - **预估工时累加算法**：基础估值表（1 文件 0.25h / 3-5 文件 1h / >10 文件 3h / 跨项目 4h） + 叠加项（设计文档 +1h / bug 文档 +1h / skill 改动 +0.5h / Rust FFI 对齐 +1h / 方案迭代 +0.5h × 轮数），单条目累计上限 8h
  - bug vs 功能判定信号表 + Mermaid 合并判定流 + 完整工作流 Step-by-step
  - 查询模式支持用户问"今天做了什么"
  - 禁区 8 条（不记对话流水、不为每次 Edit 新建条目、不在 caseflow 仓库写 `docs/work-log/` 等）
- **`CLAUDE.md`**：
  - Skill 主动触发表追加 2 行（业务源码 Edit/Write 触发 / 会话结束前强制回补）
  - Skill 索引表追加 `daily-work-log` 行，关键词含「工作日志、每日日志、工时、登记」等
- **`README.md`**：
  - 包含功能列表追加「每日工作日志」段
  - Skills 表追加 `daily-work-log` 行，明示与 `dev-log` 的分工（仓库 vs 业务项目）
- **`docs/skill-flow-20260422-v7.md`**（新建版本快照，按规范新增 skill = 较大变更）：
  - 复制自 v6
  - 头部版本信息 v6 → v7，变更摘要更新（含本次 ACL 章节扩展）
  - Skill 总览表追加 `daily-work-log` + `korepos-backend-service` 两行（后者之前未登记 skill-flow）
  - Skill 调用关系图加节点 `DWL`，连线：`JCS -. "每轮 Edit 后" .-> DWL`、`GCS -. "提交前日志已登记" .-> DWL`
  - 常见困惑速查追加 3 条：`daily-work-log vs dev-log` 区别 / 合并判定 / 工时估算
- **`.claude-plugin/plugin.json`**、**`.claude-plugin/marketplace.json`**：version `1.9.2` → `1.10.0`（**Minor** 递增，新增 skill 按规范是中位递增，非 patch）

**影响：**

- 今后在任何业务项目做代码改动时，AI 会自动（或根据用户提示）按本 skill 把当日工作沉淀到 `docs/work-log/{YYYY-MM-DD}.md`；同 bug / 同功能自动合并，不会重复建条目
- 工时估算有了可执行的规则（基础值表 + 叠加项 + 累加规则），避免 AI 凭空估值
- `dev-log` vs `daily-work-log` 分工明确：前者作用于 caseflow 仓库变更、后者作用于业务项目改动；两者**都改**的情况下**两个 skill 都要触发**
- v7 快照已生成，未来如 skill-flow 还要调整可继续在 v7 基础上迭代

## 修改 Skill · daily-work-log（补丁：首次创建 work-log 目录必须同步配置 .gitignore）

**原因：** `daily-work-log` 上线后立即在 korepos-refund 实战使用，commit 才发现工作日志文件被纳入 git 追踪；用户反馈「注意 work-log 目录配置到忽略 不要提交」。根因：skill 只规定了日志写入路径、合并规则、工时算法，**没有规定 `docs/work-log/` 是否入仓**，AI 默认 stage + commit 了第一份日志，造成清理成本（需 `git rm --cached` + 新 commit）。

**改动：**

- `skills/daily-work-log/SKILL.md`：
  - 「文件路径与命名」章节追加「⚠️ 首次创建 work-log 目录时必须配置 .gitignore（硬规则）」子节，规定 5 步流程：① Read `.gitignore` ② 检查是否已含 `docs/work-log/` 或等价规则 ③ 未含则 Edit 追加 ④ 若本会话已误 track 执行 `git rm --cached` + `chore(gitignore)` 清理 commit ⑤ 回报用户
  - 新增"已存在但格式不同"的规则匹配表（`docs/work-log/` / `work-log/` / `docs/**/work-log/` 等价匹配时跳过追加）
  - 自检清单追加一条：「首次创建 `docs/work-log/` 时，已同步检查并追加 `.gitignore`；若已误 track 过则 `git rm --cached` 清理」
  - 禁区表追加 2 条：首次创建未配置 gitignore / 把 work-log 纳入业务 commit
- `.claude-plugin/plugin.json` 与 `.claude-plugin/marketplace.json`：version `1.10.0` → `1.10.1`（Patch，补丁性质规则补充）

**影响：**

- 今后 daily-work-log 首次在任意业务项目生成 `docs/work-log/` 时，AI 会自动一并配置 `.gitignore`；若检测到已误 track（`git ls-files docs/work-log/` 非空）也会主动清理
- skill-flow v7 节点结构未变，未创建 v8 版本快照
- korepos-refund 本次已通过单独补丁 commit 完成清理：`.gitignore` 追加 `docs/work-log/` + `git rm --cached docs/work-log/2026-04-22.md`（本地文件保留）
