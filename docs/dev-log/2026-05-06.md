# 开发日志 · 2026-05-06

## 02:30 · 修改 Skill · backend-knowledge-graph-required（v1.21.1 → v1.21.2 / 范围扩展 v20.2）

**原因：** 长时间会话中识别出一个机制缺口——用户对同一技术点反复疑问、出现"为什么...还在"/"现在又卡了"等回归性措辞、修复后多轮验证追问，这些都是项目重点难点的强信号，但现有 skill 体系**没有任何**通用规则强制要求 AI 在这种情境下自动沉淀候选记录。原 `backend-knowledge-graph-required` 仅覆盖后端业务（表/SQL/状态机/原子能力），把 ffmpeg 子进程编排、并发模型、性能瓶颈、资源争夺等"非业务技术陷阱"挡在外面。`coding-violation-log` 处理的是编码规范违规，不是技术难点。`dev-log` 仅作用于 plugin 自身。结果：用户必须显式提醒"这是核心技术点要登记"才能触发沉淀，错过会话即丢失上下文。

具体促成本次决策的实战场景：在 kai-toolbox 项目中用户连续 4-5 轮追问 ffmpeg 子进程相关问题（"是不是限制了管道流"、"为什么播放卡了"、"是不是被强杀的孤儿"、"为什么 0% CPU 还占资源"），AI 解决了问题但没自动登记到知识图谱；直到用户显式提醒"这可是一个核心技术点 我的 skill 应该有提醒过吧"，AI 才补登记。这个回路不应依赖用户提醒。

**影响：**
- 今后任何项目（不只 kpay POS / korepos / 标准业务后端）的会话中，用户对同一技术点反复疑问、修复后多轮验证、出现回归性措辞，AI 必须**当场**追加候选记录到 `{ai-docs}/{project}/knowledge-graph/_candidates.md`，标记 `分类: 技术难点`，无需用户显式提醒
- 沉淀范围从纯业务（订单 / 退款 / 支付 / 表关系 / SQL）扩展到包含技术难点（ffmpeg / 子进程 / Semaphore / 缓存键 / 超时回收 / 资源争夺 / JVM 进程生命周期等）
- 不再需要新增 `tech-pain-point-tracker` 独立 skill — 复用 `backend-knowledge-graph-required` 的候选池机制（`_candidates.md` + `scenarios/` + `00_index.md`）即可
- 与 `coding-violation-log` 的边界：违规（用户纠正 AI 的写法 / 分层 / 命名）走 `coding-violation-log`；技术难点（反复疑问 / 性能 / 资源 / 子进程）走 `backend-knowledge-graph-required`
