## 一、`coding-standards-common §5.0` 规则方向反转:注释语言"沟通语言一票否决"

**原版**: §5.0 默认"沟通语言写注释",但留了"存量文件已是另一种语言 → 沿袭"的退路。

**新版**: 取消存量豁免——中文沟通的会话里新增注释一律中文,短期内单文件中英混杂可接受,后续重构再统一。唯一覆盖默认的合法路径是用户在本会话明确要求特定语言。

**为什么改**: 用户实测发现 AI 在中文会话里继续往英文文件追加英文注释,规则被退路架空。

**对应 commit**: `ca27694`(2026-05-09)

## 二、`architecture-ddd-lite-fullstack` 新增「Service 业务动作扩展强制阈值」

**原版**: "新代码落点决策"只兜底"巨型方法 / 旧骨架文件"场景,AI 看到 200-300 行 service 不命中就放行。

**新版**: 新增「Service 业务动作扩展」专节,T1-T6 六条强制拆分阈值——新状态机 / 新下游 / 新事务边界 / 新前置校验 / 既有 ≥3 个 public / 主方法 > 80 行,命中任一必须拆子 service。

**为什么改**: 用户提出场景——OrderService 已有 refund/cancel/reject,加 reverseCheckout 时 AI 惯性追加成第 4 个 public 方法,违反单一职责。需要更显式的拆分判定。

**对应 commit**: `c41fe40`

## 三、规则方向反转:T1-T6 阈值 → 「新业务分支 = 新子 service」铁律

**原版**(2 小时前):T1-T6 六条阈值 + 8 条豁免清单。

**新版**: 取消阈值与豁免框架,回到一句话原则——只要新增的是新业务分支就必须拆,无阈值无豁免。T1-T4 从"拆分阈值"降级为"识别新分支的信号",T5/T6 删除。

**为什么改**: T5/T6 + 8 条豁免清单实际成了 AI 凑判断的退路;用户指出"机械的一动作一子 service 也比留判断空间好"。

**对应 commit**: `067276e`

## 四、规则方向反转:取消"同分支变种"对 god service 的豁免

**原版**(上一小时):允许同分支参数化变种(如 `refund` 与 `partialRefund`)留在原 service。

**新版**: 取消唯一例外——同分支变种也不进 god service,要么进该分支的 focused service,要么本次扩展就是把既有方法 + 变种一并迁出 god service 的契机。god service 最多 1 行 delegate。

**为什么改**: 用户指出"共享前置状态机的情况也应抽到一个 service 里包装成两个函数",不该回到 god service。

**对应 commit**: `b28d335`

## 决策原则沉淀(给未来 AI / 维护者读)

1. **规则方向反转 = 必写 dev-log**——不只是 commit body。本次会话证明:连续反转 4 次时 commit body 已经写过的内容,从全局视角看不到"一连串决策路径",必须由 dev-log 沉淀。
2. **AI 凡是"还能写得更详细"的退路都会用到**——T5/T6 阈值 + 8 条豁免清单立刻被 AI 当成"凑判断"的工具。规则越简单越短,执行率越高。
3. **基础设施(脚本 / hook / CI)永远比文档约束可靠**——CLAUDE.md 写"必须同步 AGENTS.md"被忽视了多轮,直到加了 sync-agents.js + CI 校验才彻底解决。后续遇到"这条规则反复违反"的场景,优先考虑写脚本/hook 兜底而非加更多文字。
4. **架构原则跨语言基本一致,差异在机制实现**——focused service / 聚合边界 / 横切关注点都不分语言,差的只是 `@Transactional` vs `with session.begin():` vs `db.transaction()`。

### 决策依据(为什么本次值得 dev-log)

1. **AI 协作开发流程系统从 24 skill 长清单进化为"分层 + 显式分档"**:这是 v1.x 周期的最大结构性反转(原本"24 skill + 20 step 全跑"→ 现在"先按 S/M/L 分档跳过 70% 步骤")
2. **强制门禁正式分级**:check-design-doc hook 真启用补全 README 长期"宣称强制但脚本 disabled"的承诺与现实落差;DTO 注解 hook + git commit skill hook + 设计文档 hook 三道硬门禁(其它 skill 仍是软门禁)
3. **沉淀类 skill 转批处理**:glossary / reverse-index / daily-work-log 在 M/L 档不再中段打断,是对"沉淀价值 vs 编码体感"的明确取舍
4. **拆大 skill 不是简单的"文件物理拆分"**:每个子文档保留 H1 + 子标题 + 跨文档锚链, audit-skills.js 警告清零, check-cross-refs.js 通过, 验证拆分的健康度

- 任何 skill SKILL.md 超过 800 行 → audit-skills.js 会 warn → 必须按本次"主入口 + rules/ 子文档"模式拆分
