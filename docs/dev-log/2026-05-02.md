# 开发日志 · 2026-05-02

## 18:08 · 修改 Skill · design-doc-required

**原因：** 设计文档模板原先对完整模版固定要求多类 Mermaid 图，导致单个后端接口、退款接口这类简单场景也要补功能模块图、能力图、依赖图等低价值内容。团队真正需要的是清晰明了的设计表达：不同场景选择不同图，能一张图讲清就只画一张。

**改动：**
- 将 `design-doc-required` 的 Mermaid 规则改为“最小图原则 + 场景选图表”
- 完整模版默认要求至少 1 张核心图，其余模块图、能力图、状态图、依赖图按场景条件触发
- 轻量模版第 3 节从固定时序图调整为核心流程图，允许 flowchart 或 sequenceDiagram 二选一
- 同步更新 AGENTS、CLAUDE、README、skill-flow 与插件版本

**影响：**
- 单个后端接口/单业务动作可只画一张自身流程图或库表读写时序图
- 复杂跨模块、状态机、外部依赖场景仍保留条件图表约束

## 18:28 · 修改 Skill · design-doc-required

**原因：** 继续发现完整模版虽然已经取消固定多图，但 18 节“功能设计文档”骨架仍会诱导 AI 复写项目全集资料，例如功能模块总览、能力分解、类调用图、数据结构、下游依赖、缓存/消息/事务等。实际需要的是针对某个方案或接口开发的简明编码依据，重点确认核心逻辑与风险点。

**改动：**
- 将 `template.md` 从 18 节功能设计文档压缩为 8 节方案/接口设计文档
- 明确设计文档只写本次方案相对现状的变化和关键判断
- 强调主流程和异常流程优先合并到一张核心流程图
- 将数据与依赖变更、调用关系、模块交互等改为“只写本次变化，不复写项目全集资料”

**影响：**
- 完整模版也变成简明编码依据，不再默认要求模块级大设计
- 简单接口、单个后端动作、已有模块内方案开发的设计文档更短、更聚焦

## 18:40 · 修改 Skill · backend-knowledge-graph-required

**原因：** 后端接口开发中反复出现“订单场景改了哪些表、表状态如何变化、部分退如何判定、订单业务状态如何计算”等问题，但现有后端知识图谱只沉淀能力/流程/表/枚举，没有形成编码前回顾、编码中复用、编码后回写的表逻辑闭环。

**改动：**
- 将后端知识图谱从 Java 单服务扩展为后端单服务业务图谱，重点覆盖表逻辑、状态判定和原子能力复用
- 新增 `07_table_logic_index.md`、`08_atomic_capability_index.md`、`table-logic/`、`atomic-capabilities/` 目录规范
- 规定后端接口编码前必须回顾表逻辑索引和原子能力索引，已有能力优先复用
- 规定编码后将 DAO/SQL、订单/退款/支付状态判定、金额聚合、表状态变更同步到正式图谱或用户目录候选池

**影响：**
- 新增接口前可以基于表逻辑图谱判断现有表和原子能力是否支持
- 订单部分退、订单状态判定等反复业务问题会被沉淀为可复用知识，而不是每次重新问代码

## 19:05 · 修改 Skill · bugfix-coding-style

**原因：** 之前规则已经禁止 `[BUGFIX]` / `[DEPRECATED]` / 日期标记等变更历史注释，但仍可能诱导 AI 把旧实现缺陷、新实现步骤、设计文档摘要、未来版本计划整段堆在函数头，导致源码注释像变更复盘，影响阅读。

**改动：**
- 明确这类注释归 `bugfix-coding-style` 约束，不另起新 skill
- 规定函数 / 类 doc comment 只写当前职责、输入输出语义、不变式和误用风险
- 禁止函数头出现 `[REWRITTEN 日期]`、旧实现复盘、实现步骤流水、未来版本计划和设计文档摘要
- 规定复杂逻辑说明贴近对应代码块，用 1-2 行 WHY 注释表达

**影响：**
- 源码更接近“当前正确逻辑说明”，不再承载历史版本故事
- 复杂退款、订单、分摊等逻辑仍允许注释，但注释必须短、局部、服务当下阅读

## 19:18 · 修改 Skill · korepos-backend-service

**原因：** korepos / korepos-refund 后端接口开发中出现存量违规和重复能力容易被继续扩散的问题：Service 裸 SQL、长方法堆逻辑、DB 状态值写裸数字，以及跨 feature 的业务计算没有稳定复用入口。

**改动：**
- 新增编辑 backend 代码前的违规自检前置，要求先扫裸 SQL、长方法、DB 字段裸数字、裸 Exception 并汇报处置
- 新增 Service 长方法必拆 `_xxxStep` 私有方法规则，明确私有 step / internal / backend_infra services 的升级梯度
- 新增 `common/backend_infra/services/` 跨 feature 业务原子能力层和 `INDEX.md` 索引维护规则
- 新增 DB 字段值与枚举绑定规则，禁止状态、类型、标志位等字段使用裸数字字面量

**影响：**
- 新写 backend service 前必须先查可复用业务原子能力，减少重复 SQL 和重复业务计算
- 存量违规不会在新接口开发时被无声复制，长方法和魔法数字会被前置暴露并受控处理

## 20:08 · 修改 Skill · backend-knowledge-graph-required

**原因：** 实战中发现 skill 触发存在两大盲区，导致后端业务知识在会话中流失：

1. **被动触发**：当前 description 偏向"用户主动要求建图谱"或"项目已有 docs/knowledge-graph/backend/"。AI 在调查云端表关系（如 1bill+Ntx、退款 item 是新建快照还是引用原表、分摊按 tx 粒度比例算）后**直接把结果写成一份 Markdown 沉淀**，全程未触发本 skill；既错过候选池入栈，也没经过本 skill 决定文档归属和模板。
2. **跨项目误判**：当前主工作目录是项目 A（前端 Flutter），调查对象是项目 B（云端 Java 单服务）时，AI 容易把这种场景误判为"跨项目"转给 `cross-project-locator`；实际"云端单服务内部的表关系"始终属于本 skill，与 cwd 无关。
3. **入门门槛过高**：原 skill 列了 8 份索引文件 + 6 个子目录，让"还没建图谱"的项目觉得太重，最终连 Tier 1 都没建，直接跳过 skill 写散文。

用户原话定位需求：知识图谱本质是"业务场景关系对应的数据库表扭转关系"，通过多个细小场景汇总成全局后台逻辑；编码中经常会问，必须能自动落地、整理；写新业务场景时要能快速回顾分析是否与现有冲突。

**改动：**

- **frontmatter description 重写**，列出 6 类 BLOCKING 触发场景：
  - 即将 Write/Edit 任何描述后端表关系/ER/状态扭转/业务流程→DB CRUD 的 .md（无论路径，含 `ai-docs/`、`work-log/`、`scenarios/` 而不仅 `docs/`）
  - 用户问"X 与 Y 是 1:1 还是 1:N" / "改这个动哪些表" / "字段从哪来" / "退款/账单/流水/分摊怎么算" / "是新建快照还是引用原表"等表关系/状态/原子能力问询
  - AI 完成后端调查并发现 ≥1 条可复用事实——含跨项目以另一服务为调查对象的场景
  - 用户主动要求初始化/读/更新图谱
  - 项目已有 `docs/knowledge-graph/backend/`
  - 后端代码变更
- **新增「BLOCKING 强触发清单」章节**：以表格列出 6 类触发信号 + 必做动作；附"常见误判反例"列出已经踩过的坑（"用户只是问云端逻辑不用触发"、"本地工程没图谱目录所以 skill 不适用"、"主目录 A 调查 B 应交给 cross-project-locator"、"我直接 Write 一份图谱文档就行了"全部明确为错）。
- **新增「调查对象与图谱归属路由」章节**：明确**图谱归属永远 = 被调查的后端单服务项目，与 cwd 无关**；候选池路径 `ai-docs/{被调查项目}/knowledge-graph/_candidates.md`；正式图谱仅在用户明确要求"上传终版到项目内"才走 `{被调查项目根目录}/docs/knowledge-graph/backend/`；同时给出本 skill 与 `cross-project-locator` 的边界判定（单服务内部归本 skill，服务间链路归 cross-project-locator，可同时触发）。
- **「推荐目录」章节改写为渐进式三层骨架**：
  - **Tier 1 起步**（必须，最小可用）：只要 `00_index.md` + `scenarios/{业务场景}.md` + `_candidates.md` 即可投入使用；场景小卡固定模板（一句话总述 + 一张关系图 + 表/字段速查表 + 核心规则带代码坐标 + 已知疑点 + 交叉引用）；明确"第一份图谱只要 1 份场景卡 + 1 份索引就算建立"
  - **Tier 2 成熟**（≥3 个场景后再分）：tables / enums / flows / atomic-capabilities / 07_table_logic_index / 08_atomic_capability_index
  - **Tier 3 全景**（≥10 个场景或正式发版前再补）：service profile / domain capability map / data model map / enum dictionary / api entrypoints / external dependencies / business flow index
- **候选池路径精简**：旧版 `{agent}/{YYYY-MM-DD}/backend-kg-candidates.md` 改为 `_candidates.md`（按被调查项目而非按日期分），便于跨日期跨会话累积同主题事实并最终归并到 scenarios 卡；旧路径仅作向后兼容、不再推荐。
- **强化"自动追加"语气**：AI 回答任何后端表关系/业务判定/状态扭转/字段语义问题，**即便用户没要求"建图谱"**，回答的同一回合内必须自动追加 1 条候选记录到 `_candidates.md`；错过即流程违反。给出固定记录格式（时间 / 场景 / 事实 / 涉及表 / 来源 / 可信度 / 后续动作）。
- **「分析前读取顺序」章节**适配 Tier 1/2/3 分层：Tier 1 阶段只读 `00_index.md` + 命中场景小卡 + `_candidates.md`，不强求 Tier 2/3 索引存在。
- **CLAUDE.md 主动触发表第 24 行**：替换为强语气版（写前拦截 / 问询触发 / 调查对象路由 / 候选自动沉淀全部点出）。
- **CLAUDE.md Skill 索引第 56 行**：覆盖范围/关键词补 `v1.18.7` 起的关键变化标记（写前拦截、问询触发、调查对象路由、被调查项目归属、cwd 无关、渐进式三层、Tier 1/2/3、scenarios 场景小卡）。
- **`.claude-plugin/plugin.json` 与 `.claude-plugin/marketplace.json` 版本同步升至 1.18.7**（按 SemVer 表"修复 Skill 内容、调整措辞、补充规则" → Patch）。
- **docs/skill-flow.md** 头部追加 v19.8 摘要（同 v19.5 backend-kg 强化的处理风格，链路图节点和连线无变化，按"轻微"未单独建快照）。

**影响：**

- AI 回答后端单服务表关系类问题时不再出现"答完就走、事实散落"的盲区；同一回合必须候选沉淀。
- "主目录 ≠ 调查对象" 的场景明确归属：图谱归被调查项目，不再被错误转给 `cross-project-locator`。
- 项目从零启动图谱的入门门槛降到 Tier 1 三个文件；"细小场景汇总成全局" 的设计目标通过场景小卡 + 索引反查表的方式自然达成。
- 本次会话已发生的踩坑（直接 Write `云端退款数据模型_知识图谱.md` 未经本 skill）已被 BLOCKING 触发清单与"常见误判反例"覆盖，避免再次发生。
- 候选池统一收口到 `ai-docs/{被调查项目}/knowledge-graph/_candidates.md`，跨会话累积同主题事实；旧路径降级为向后兼容，不影响存量。

**与其他 skill 的关系：**

- 与 `cross-project-locator` 的边界更清晰：单服务内部 = 本 skill；服务间链路 / 接口对照 / 调用映射 = `cross-project-locator`；同一会话两者可串行触发。
- 与 `doc-index-required` 的边界保持原有：`ai-docs/` 草稿不触发 doc-index，仅在用户明确要求写入项目 `docs/` 时由 doc-index 接管。
- 与 `markdown-writing-standards` 的边界保持原有：场景小卡里的 mermaid 图按 markdown-writing-standards 规则写。
