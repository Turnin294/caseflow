## 修改 Skill · korepos-backend-service（v1.12.0 → v1.13.0，新增 Service 粒度规则）+ 补登 CLAUDE.md 索引

**原因：** 实践中观察到 backend service 文件越写越胖 —— 单一 service 类挂 3-5 个 public 方法对应不同 endpoint（`refund_query_service.dart` 同时 expose `getMaxAmount` / `getAllocations` / `getProducts`），debug 时栈与日志混在一起、版本控制 diff 互相牵连、未来按 endpoint 独立服务化时调用图爆炸。根因是同事把"本端 service"按云端 Spring `ServiceImpl` 心智模型来写（一个 Impl 装多个业务方法），但本端 service 实际语义角色 = 云端 `Controller`（HTTP 入口的薄编排层），二者合并的合理性（DI / 跨方法事务 / Spring AOP）在本端都不成立。需要在 skill 里把心智模型校准并下硬规则。

另外发现 `korepos-backend-service` skill 自创建以来**从未在 `CLAUDE.md` 索引表登记过**（既不在主动触发表也不在 Skill 索引表），团队成员看 `CLAUDE.md` 索引根本发现不了这个 skill 的存在，本次一并补登。

**改动：**

- `skills/korepos-backend-service/SKILL.md`：
  - 目录结构表 service 文件命名 `{module}_{feature}_service.dart` → `{module}_{action}_service.dart`，并在注释里指向新「Service 粒度规则」节
  - 新增整节「Service 粒度规则：一接口一 service（debug 友好原则）」(60+ 行)：
    - 心智模型校准对照表（云端 Java 各角色 ↔ 本端 Flutter backend 各角色）
    - 6 条强制规则：粒度（1 文件 = 1 endpoint）/ 类内单一 public 方法（= handler 转发方法名）/ 文件命名（`{module}_{action}_service.dart`）/ 跨接口复用走 `service/{purpose}_orchestrator.dart` / service 之间禁止互相 import / DTO 自闭环不变
    - 反例 → 正例（`refund_query_service.dart` 1:N 拆 `get_max_refundable_amount_service.dart` / `get_refund_allocations_service.dart` / `get_refund_products_service.dart`）
    - 命名一致性收敛表：动作型 `{action}_{resource}_service.dart` 允许新增；域型 `{module}_{domain}_service.dart` 不允许新增
    - 存量 1:N 文件保持不动，由专项 PR 拆分，本规则仅对**自落入 SKILL.md 之日起新增**的 service 文件强制生效
  - 「Service 编写步骤」section 顶部规则补「一接口一 service」要点 + 跨接口复用走 orchestrator 的硬约束
  - 自检清单新增 1 行「service 粒度」检查项（每个 service 文件只对应 1 个 endpoint / 类内只暴露 1 个 public 方法 / 跨接口复用沉入 orchestrator / service 之间无互相 import）
  - 常见红线表新增 2 条（一个 service 文件 expose 多个 public 业务方法 / service A 直接 import service B 复用业务能力）
- `CLAUDE.md`：
  - 主动触发规范表新增 1 行：写 korepos / korepos-refund 后端接口、`lib/features/{module}/backend/` 路径下 Edit/Write、用户说「加接口 / 加 endpoint / 写 backend 服务 / 按 UI 对接手册实现接口」时触发 `korepos-backend-service`
  - Skill 索引表新增 1 行：覆盖范围涵盖目录结构 / BackendInfra 门面 / UI 层泄漏拦截 / DTO 自闭环 / **一接口一 service 粒度规则** / 8 步编写顺序 / 自检清单 / 红线，关键词带 `一接口一 service / orchestrator / 粒度 / ACL 防腐蚀`
- `plugin.json` / `marketplace.json` 版本 1.12.0 → 1.13.0（Minor）—— 实质性强制规则补充 + 新增命名约定 + 索引补登，按字面归 Patch 偏轻，按行为变化归 Minor 更贴切

**影响：**

- 新写的 backend service 文件**强制走「一接口一 service」**：1 文件 = 1 endpoint = 1 public 方法（= handler 转发方法名），文件名 `{module}_{action}_service.dart`
- 跨接口复用链路必须沉到 `service/{purpose}_orchestrator.dart`（典型如取消订单的 `whole_order_cancel_orchestrator.dart` 模式），**严禁** service A 直接 import service B
- 存量 1:N service 文件（`refund_query_service.dart` 等）**保持不动**，等专项 PR 拆分；改动现有文件时若机会成熟可顺手拆，但不要为了拆而开 PR
- 团队成员 `/plugin update` 后能从 `CLAUDE.md` 索引表里看到 `korepos-backend-service` 的存在；之前漏登记的盲区消除
- 第一个验证用例：refund-backendv2 后续新增的接口（如 ⑩ ⑪ ⑫ 系列）一律按新粒度规则写，不再混入既有 `refund_query_service.dart`
