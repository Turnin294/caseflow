# 开发日志 · 2026-04-30

## 18:30 · 方向反转 · bugfix-coding-style v1.17.0

**原因：** 在 `korepos-refund` 项目修复一个 POS 退款回调反查 bug 时，按当时的 `bugfix-coding-style` 规范在 DAO 内保留了 `[BUGFIX 2026-04-30]` 头注释 + 整段 `//` 注释保留的旧 SQL，方法体里堆了 ~20 行变更日志叙事。用户明确反馈这种"把变更历史写进代码"的做法噪声大、对未来读者无价值，并要求把规范沉淀到 caseflow plugin 中：以后代码只保留正确的逻辑说明，且尽量上提到方法/类的 doc comment。

**改动：**
- 反向重写 `skills/bugfix-coding-style/SKILL.md`：
  - 废止 A 类（`[DEPRECATED YYYY-MM-DD]` + 注释保留旧代码）规则
  - 废止 B 类（`[ADDED YYYY-MM-DD]` 头注释）规则
  - 新增红线清单：禁止 `[BUGFIX]` / 日期标记 / `// 详见 vN 调整流水` / `// PR #1234` / 大段注释保留旧代码 / 个人化 TODO 出现在源码中
  - 新增"WHY 注释优先上提到方法/类 doc comment"原则，并给出推荐写法对照
  - 适用范围扩展到所有源码改动，不再局限于联调期/轻量修订流水
  - 调整旧标记清理时机：编辑同一段代码时可顺手清理 `[DEPRECATED]` / `[ADDED]`（与旧版"AI 永不主动删"规则相反）
- 更新 `CLAUDE.md` 主动触发表第 27 行（联调期 → 所有源码改动；规则关键词反转）
- 更新 `CLAUDE.md` Skill 索引表第 63 行（描述、关键词整体反向）
- 更新 `docs/skill-flow.md` 4 处：头部"最后更新"升到 v16.6 + 变更摘要 / 第 52 行索引行 / 第 173 行流程图节点文字 / FAQ 第 294-295 两条问答
- `.claude-plugin/plugin.json`、`.claude-plugin/marketplace.json` 版本号：1.16.5 → 1.17.0（规则方向反转，按 Minor）

**影响：**
- 今后所有 bug 修复 / 对齐云端 / 删冗余 / 联调期改动 / 普通业务代码改动都遵循新规范：直接改写、不留变更历史
- 变更原因的归属：commit message body / bug 文档 / 设计文档；不入源码
- 旧版本（v1.16 及以前）按规范留下的 `[DEPRECATED]` / `[ADDED]` 注释段，不主动批量清理；改到同段代码时顺手清掉即可
- 与 `coding-violation-log` 的协作关系不变：用户后续若发现 AI 又写了 `[BUGFIX 日期]` 标记，由 coding-violation-log 登记防重犯
