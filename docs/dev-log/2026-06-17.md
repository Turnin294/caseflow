# 2026-06-17 · SQL→DDL 前置校验补全（check-sql-ddl-readiness + ddl-baseline 落地）

## 背景（真实踩坑）

Yoooni（Java + Struts2 + iBatis + Oracle）改打版进度工作台 SQL 时，把「染厂收胚时间」误取 `tdate`，绕了好几轮才改回正确口径 `transfer.checkdate`——而项目知识图谱里**早就**写明了 `makedate`=配送染厂、`checkdate`=染厂收胚（场景卡「打版送染厂调拨关联」+ `_sql_candidates.md`）。即规则、口径全都对，是「写 SQL 前没去核对」这一步没落地。

## 根因（为什么既有护栏没拦住）

1. **`check-backend-kg-readiness.js` 路径白名单是 Dart/Flutter 专用**（`lib/features/**/backend/**/*.dart`、`lib/common/backend_infra/...`）。Yoooni 的 iBatis `Morder.xml` / `.sql` / `*Dao.java` 完全不在白名单内 → hook 对该项目从不触发。
2. **该 hook 豁免 ≤20 行小改**，而 `tdate → checkdate` 是 1 行字段名改动——正是最该拦、却被豁免的那类。
3. 该 hook 把「读过任意一张图谱卡（00_index / scenarios / ddl-baseline）」都算满足，没有「写 SQL 必须读 DDL 基线」这个专门约束。

`backend-knowledge-graph-required` SKILL 文本其实已重度要求 `ddl-baseline.md`（Tier1 必备 + 专节 + 编码前第 1 步），但**机械兜底层对非 Dart 栈是空白**。

## 决策

新增一道**职责单一、语言无关**的 PreToolUse hook，而不是去拓宽 Dart 专用的那道（避免把两个关注点揉在一起、影响既有 Flutter 项目行为）：

- `hooks/check-sql-ddl-readiness.js`：按「文件是否承载 SQL」判定（iBatis/MyBatis SqlMap XML / `.sql` / `*Dao`·`*Mapper`·`*Repository` / 含裸 SQL·ORM raw query 的源码），命中且本会话未 Read 过 `ddl-baseline.md` → stderr 提示。
- **关键差异**：① 不绑定技术栈目录；② **不豁免小改**（1 行 SQL 字段名改动也拦，这正是本次踩坑形态）；③ 专门要求读 `ddl-baseline.md`（字段级权威源），而非「读过任意图谱卡」即可。
- 默认 `warn`（exit 0 + stderr，先评估误报率），`TEAM_STANDARDS_SQL_DDL_HOOK=block|off` 调档；与既有 hook 一致的渐进上线策略。

同步：
- SKILL `backend-knowledge-graph-required` 触发信号补 iBatis/MyBatis XML、Java/Kotlin DAO、`.sql`、Oracle；新增「无论改动大小，改 SQL 前必读 ddl-baseline」一行；「hook 集成」节说明两道 hook 的分工。
- 为 Yoooni 落地 `knowledge-graph/ddl-baseline.md`（先种子占位 + Oracle dump 脚本，已种入 `erp_transfer`/`erp_morder` 家族及 `makedate`/`checkdate` 语义），并在 `00_index.md` 置顶「写 SQL 前必读」。

## 边界（为什么 DDL 基线不进 domain-knowledge MCP）

DDL（字段名/类型/默认值/索引）是**实现级事实**，会随 schema 迁移变化，按 `project-domain-knowledge` 的 §0 边界（「业务域真理」进、「当前怎么实现」不进）**不进业务认知库**。它归项目级知识图谱（`ai-docs/{project}/knowledge-graph/`），由 team-standards 的 skill+hook 管控。MCP 侧无改动。

## 追加澄清：实现级知识（DDL/SQL/ER）的权威源 = 集中库 project-domain-knowledge，不是个人 ai-docs，也不是各业务项目 docs/

落地 Yoooni `ddl-baseline.md` 时一度只放进个人 ai-docs（草稿区）就收工，被用户纠正"每人一份会漂移"。我先改放进 Yoooni 项目仓 `docs/knowledge-graph/`——又被纠正：那仍是"落到具体本项目"，团队设计是**集中在我们的知识库/MCP 里、专人统一维护**。

最终（用户拍板）：单项目实现级知识的**中心权威源 = `project-domain-knowledge` 仓的实现知识区 `knowledge/{project}/impl/`**，由 domain-knowledge MCP 生态承载、owner 统一维护、全员 git/MCP 同步。这需要扩展 PDK 边界（原 §3「实现细节不进」）——做法是**分区隔离**：业务真理区 `{project}/{module}/` 不变、永不被 DDL 污染；新增实现知识区 `{project}/impl/`，大资产（全量 DDL dump）不带 `id` 不入索引按仓库资产分发、小导引卡入索引可 MCP 检索。`cross-topology` 仍只管跨 ≥2 项目。

动作：① Yoooni 真实 `ddl-baseline.md`(1109 表) 落 `project-domain-knowledge/knowledge/yoooni/impl/`（资产）+ `ddl-guide.md`（导引卡，已验证 MCP `search("DDL")` 命中）；② 撤回错放的 `yoooni/docs/knowledge-graph/` 与 ai-docs 的 887KB 重复副本；③ 同步改 PDK `_taxonomy.md` §0/§1.5 + 设计文档 §2/§3（分区边界）、ai-docs 索引指向 PDK、本 skill「DDL 基线」节的权威源红线。

## 追加：禁止 AI 署名从「SKILL 规则」升级为「机械红线」（1.37.0）

### 背景

宿主/IDE 默认指令要求每条 commit 结尾追加 `Co-Authored-By: Claude <noreply@anthropic.com>`。git-commit-standards 早已明令禁止任何 AI 署名（§Author「只填真实提交者」、§Red Flags「加 Co-Authored-By 立即删除」），但那是 **SKILL 层规则**：小改动（≤2 文件 ∧ ≤30 行 ∧ 全 `M`）不触发 git-commit-standards 完整流程，宿主默认签名就被默认带上、无人拦截。规则在、强制缺。

### 决策

加第 6 道 PreToolUse(Bash) hook `check-commit-no-ai-signature.js`：在 `git commit` 落盘前机械扫描提交信息（`-m` 内联 / heredoc / `-F` 文件），命中 AI 署名 trailer 即 `exit 2` 阻断，**与改动大小、是否走 skill 完全无关**。这是把"项目侧 team-standards 规定优先于宿主通用默认"落到机械层——不靠模型自觉记得删。

设计要点：
- 只匹配**署名形态**（`Co-Authored-By:` 行指向 claude/anthropic/copilot/cursor/codex/…、`noreply@anthropic.com`、机器人 emoji、`Generated with/by <AI>` 落款），正文普通提及 "claude"（如 "适配 claude api 超时"）**不误伤**——自测覆盖。
- 默认 `block`（客观红线，同 check-comment-density 的客观项策略）；`TEAM_STANDARDS_AI_SIGNATURE_HOOK=warn|off` 可调。
- 局限：编辑器交互式 commit（无 `-m/-F`）PreToolUse 拿不到正文，不覆盖；但 AI 几乎都用 `-m`，威胁面已堵住。

### 版本

team-standards 1.36.0 → **1.37.0**（MINOR：新增机械 hook）；三处 manifest（plugin.json ×2 + marketplace.json）同步。
